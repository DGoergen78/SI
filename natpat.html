Grundlagen

Was ist NAT?

NAT steht für Network Address Translation – also die Übersetzung von IP-Adressen. Ein Router ändert dabei die Quell- oder Ziel-IP-Adresse eines Pakets, bevor er es weiterleitet.

💡 Das Kernproblem

Im Internet gibt es nur ~4,3 Milliarden IPv4-Adressen – viel zu wenig für alle Geräte weltweit. Deshalb gibt es private IP-Adressen (nur im lokalen Netz) und öffentliche IP-Adressen (im Internet). NAT ist die Brücke zwischen beiden Welten.

Private IP-Adressbereiche (RFC 1918)

10.0.0.0/8    → Class A privat (z.B. 10.0.0.1 – 10.255.255.254)
172.16.0.0/12 → Class B privat
192.168.0.0/16 → Class C privat

Diese Adressen sind nicht im Internet routbar. Jedes Heimnetzwerk oder Firmennetz kann sie frei verwenden – ohne Konflikte mit anderen Netzen.

Wie funktioniert NAT grundsätzlich?

Client (privat)
10.0.0.1
NAT-Router
ändert IP ↔
Server (öffentlich)
12.7.51.9

Der Router tauscht die private Quelladresse gegen seine öffentliche IP aus. Nach außen sieht es so aus, als käme alles vom Router selbst.

  • Eingehend: Client schickt Paket mit privater IP (10.0.0.1) → Router ersetzt sie durch öffentliche IP

  • Speicherung: Router merkt sich in einer NAT-Tabelle, welche private IP welchem ausgehenden Paket entspricht

  • Antwort: Antwortpaket kommt zurück zur öffentlichen IP → Router schlägt in der Tabelle nach und schickt das Paket zur privaten IP weiter

Grundlagen

Von NAT zu PAT

⚠️ Das Problem mit einfachem NAT

Reines NAT (1:1) würde bedeuten: Ein Client = Eine öffentliche IP. Das löst das Adressproblem kaum. Was, wenn 50 Clients gleichzeitig surfen sollen und man nur eine öffentliche IP hat?

PAT = Port Address Translation

PAT (auch NAT Overloading oder Masquerading) löst dies durch den Einsatz von Port-Nummern. Mehrere private IPs werden auf eine öffentliche IP abgebildet – aber jede Verbindung bekommt einen eigenen Port-Eintrag.

Was sind Ports?

Ports (1–65535) identifizieren Dienste und Verbindungen innerhalb eines Computers:

Port 80 → HTTP (Webseiten, unverschlüsselt)
Port 443 → HTTPS (Webseiten, verschlüsselt)
Port 22 → SSH (Fernwartung)
Port 53 → DNS (Namensauflösung)

Wenn dein Browser eine Verbindung aufbaut, wählt das Betriebssystem einen zufälligen Ephemeral-Port (>1024) als Quell-Port. Im Beispiel: 45123

PAT in Aktion – Zwei Clients, eine öffentliche IP

Intern (privat)Extern (öffentlich)
SourceDestinationSource (nach NAT)Destination
10.0.0.1:45123 12.7.51.9:443 86.14.30.2:10001 12.7.51.9:443
10.0.0.2:45123 12.7.51.9:443 86.14.30.2:10002 12.7.51.9:443

Beide Clients haben zufällig den gleichen Quellport 45123 gewählt. Der Router löst den Konflikt durch unterschiedliche öffentliche Ports.

✅ Vorteile von PAT

  • Hunderte Clients teilen sich eine einzige öffentliche IP
  • Private Netzwerk-Topologie bleibt von außen unsichtbar
  • Implizite Firewall: Unaufgeforderte eingehende Pakete werden geblockt
  • Standard in fast jedem Heimrouter (Fritz!Box, etc.)

Analyse: Quell-Ports

Beide Clients haben zufällig denselben Ephemeral-Port 45123 gewählt. Das ist kein Fehler – das Betriebssystem wählt Ports zufällig aus dem Bereich 1024–65535.

Problem: Wenn der Router beide Pakete mit der gleichen öffentlichen IP weiterleitet und beide den Port 45123 nutzen – wie unterscheidet er dann die Antworten?

PAT-Lösung: Port-Remapping

Der NAT/PAT-Router weist jedem ausgehenden Paket einen eindeutigen öffentlichen Port zu. Er führt dazu eine PAT-Tabelle:

Privat (intern)Öffentlich (extern)Zweck
10.0.0.1:45123 31.101.17.41:10001 Client 1 – HTTPS
10.0.0.2:45123 31.101.17.41:10002 Client 2 – HTTPS

Die öffentlichen Ports 10001/10002 sind zufällig gewählt (Router-intern). Was zählt: Sie sind eindeutig.

✅ Abschnitt 2: Nach der NAT-Übersetzung (Lösung)

Source (öffentlich)Destination
31.101.17.41:1000112.7.51.9:443
31.101.17.41:1000212.7.51.9:443

Der Server sieht: Zwei Verbindungen von der gleichen IP, aber verschiedenen Ports. Die privaten Adressen sind komplett verborgen.

🔑 Warum Port 443?

Port 443 ist der Standard-Port für HTTPS (HyperText Transfer Protocol Secure). Die Aufgabe sagt „eine Webseite im Internet aufrufen" – moderne Webseiten nutzen fast immer HTTPS. Deshalb ist das Ziel 12.7.51.9:443.

NAT/PAT Schritt 2

Abschnitt 3: Die Antwort des Servers

Der Server im Internet hat die Anfragen empfangen und sendet nun seine Antwortpakete zurück. Welche Adressen stehen drin?

💡 Logik der Umkehrung

In der Netzwerkommunikation gilt: Was in Abschnitt 2 die Source war, wird in Abschnitt 3 die Destination – und umgekehrt. Der Server antwortet einfach zurück an die Adresse, von der er die Anfrage erhalten hat.

Was der Server weiß

Der Server hat zwei Pakete erhalten, beide von:

Paket 1: von 31.101.17.41:10001
Paket 2: von 31.101.17.41:10002

Der Server weiß nichts von 10.0.0.1 oder 10.0.0.2. Er antwortet an die öffentliche IP mit den entsprechenden Ports.

✅ Abschnitt 3: Antwort vom Server (Lösung)

Source (Server)Destination (zum Router)
12.7.51.9:44331.101.17.41:10001
12.7.51.9:44331.101.17.41:10002

Warum antwortet der Server auf Port 443?

Der Server-Prozess (z.B. ein Webserver) lauscht auf Port 443. Die Antwort kommt also von 12.7.51.9:443 – das ist der Source-Port der Antwort.

Das ist wichtig: Port 443 wechselt hier von Destination (in Abschnitt 2) zu Source (in Abschnitt 3).

NAT/PAT Schritt 3

Abschnitt 4: Rückkehr ins LAN

Die Antwortpakete kommen beim NAT-Router an. Jetzt muss er sie an die richtigen Clients im LAN weiterleiten.

Der Router schlägt nach

Der Router empfängt Pakete für 31.101.17.41:10001 und 31.101.17.41:10002. Er schaut in seiner PAT-Tabelle nach:

Öffentl. Port→ Private IP:Port
31.101.17.41:1000110.0.0.1:45123
31.101.17.41:1000210.0.0.2:45123

Er tauscht Destination gegen die private IP:Port aus – und leitet das Paket weiter.

✅ Abschnitt 4: Im LAN (Lösung)

Source (Server)Destination (Client)
12.7.51.9:44310.0.0.1:45123
12.7.51.9:44310.0.0.2:45123

Jetzt sieht im LAN jeder Client seine eigene Adresse als Ziel – als hätte der Server direkt mit ihm gesprochen.

NAT/PAT-Prozess in 4 Sätzen erklärt

  1. Ein Client im privaten Netz sendet ein Paket mit seiner privaten IP und einem zufälligen Quellport an den Server.
  2. Der NAT/PAT-Router tauscht die private Quelladresse gegen seine öffentliche IP und einen eindeutigen Port aus und merkt sich die Zuordnung in seiner PAT-Tabelle.
  3. Der Server antwortet an die öffentliche IP:Port des Routers – der Router schlägt in seiner Tabelle nach.
  4. Der Router leitet das Antwortpaket an den richtigen privaten Client weiter und stellt so die korrekte Zustellung sicher.

Port-Bereiche

Ports sind 16-Bit-Zahlen (0–65535) und in drei offizielle Bereiche eingeteilt. Für NAT/PAT sind besonders Well-Known Ports (Zielport des Servers) und Ephemeral Ports (Quellport des Clients) relevant.

🔎 Relevanz für die Prüfungsaufgabe

Port 443 → Well-Known · Zielport des HTTPS-Webservers (12.7.51.9)
Port 45123 → Ephemeral · Quellport der Clients, zufällig vom OS gewählt
Hinweis: Port 45123 liegt im Linux-Ephemeral-Bereich (32768–60999), nicht im Windows-Bereich (49152–65535). Für die IHK-Prüfung gilt: jeder Port >1023, der zufällig vom OS vergeben wird, ist ein Ephemeral Port.

⚠️ Warum können zwei Clients denselben Ephemeral Port wählen?

Ephemeral Ports werden pro Gerät vergeben – jedes Betriebssystem verwaltet seinen eigenen Port-Pool unabhängig. Es ist also völlig normal, dass Client 1 und Client 2 zufällig denselben Port 45123 wählen. Der NAT/PAT-Router erkennt den Konflikt und löst ihn durch Zuweisung unterschiedlicher öffentlicher Ports.
