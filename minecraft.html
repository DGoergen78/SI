Minecraft Server im Heimnetz

Du willst einen eigenen Minecraft Server betreiben, damit Freunde von außen mitspielen können. Das klingt einfach – aber zwischen deinem Server und deinen Freunden stehen Router, NAT, Firewalls und das DNS-System.

🌍 Das Grundproblem

Dein Minecraft Server hat eine private IP (z.B. 192.168.1.100) – die ist im Internet nicht erreichbar. Dein Router hat eine öffentliche IP, aber die ändert sich bei den meisten Heimanschlüssen täglich. Und selbst wenn Freunde die richtige IP kennen: Der Router weiß nicht, an welches Gerät er Minecraft-Pakete weiterleiten soll.

Was muss gelöst werden?

  • Erreichbarkeit: Der Router muss wissen, dass eingehende Pakete auf Port 25565 an den Server-PC weitergeleitet werden sollen → Port-Weiterleitung

  • Adressierbarkeit: Freunde brauchen eine feste Adresse. Da die öffentliche IP sich täglich ändert, braucht man einen festen Hostnamen → DynDNS

  • Freigabe: Router-Firewall und PC-Firewall müssen den Minecraft-Port erlauben → Firewall-Regeln

  • Stabilität: Der Server-PC braucht eine feste private IP, damit die Port-Weiterleitung immer auf das richtige Gerät zeigt → DHCP-Reservierung oder statische IP

Gesamtbild: Was passiert wenn ein Freund verbindet

Port 25565 – Der Minecraft-Standard

Minecraft Java Edition verwendet TCP Port 25565. Dieser Port muss im Router weitergeleitet und in der Firewall freigegeben werden. Bedrock Edition nutzt UDP Port 19132.

Schritt 1

Port-Weiterleitung

Die Port-Weiterleitung (engl. Port Forwarding) ist eine statische NAT-Regel: Eingehende Pakete auf einem bestimmten Port werden immer an ein bestimmtes Gerät im LAN weitergeleitet.

💡 Warum reicht normales PAT nicht?

Normales PAT/NAT ist ausgehend: Der Client im LAN baut eine Verbindung auf, der Router merkt sich die Zuordnung. Dein Minecraft Server wartet aber passiv auf eingehende Verbindungen – es gibt keinen Client, der zuerst nach außen verbindet. Der Router hat also keinen Tabelleneintrag und weiß nicht, wohin er das Paket schicken soll. Port-Weiterleitung löst das durch eine feste, dauerhafte Regel.

PAT (ausgehend) vs. Port-Weiterleitung (eingehend)

So richtest du Port-Weiterleitung ein

Im Router-Menü (z.B. Fritz!Box: Heimnetz → Netzwerk → Portfreigaben) legst du folgende Regel an:

Protokoll: TCP Externer Port: 25565 Interne IP: 192.168.1.100 ← dein Server-PC Interner Port: 25565

Ab jetzt leitet der Router jedes eingehende Paket auf Port 25565 immer an 192.168.1.100 weiter – egal ob gerade jemand verbunden ist oder nicht.

Eingehende Pakete: Wohin leitet der Router sie?

⚠️ Feste private IP ist Pflicht!

Die Port-Weiterleitung zeigt auf eine bestimmte IP-Adresse. Wenn der Server-PC per DHCP eine neue IP bekommt, geht die Weiterleitung ins Leere. Lösung: Im Router eine DHCP-Reservierung (MAC-Binding) einrichten, oder am Server-PC eine statische IP vergeben.

Schritt 2

DynDNS – Dynamisches DNS

Dein Heimanschluss bekommt vom Internetanbieter (ISP) in der Regel keine feste öffentliche IP – sie ändert sich meist täglich. Freunde könnten heute 91.48.77.200 eingeben, morgen wäre die IP schon eine andere.

🌐 Was ist DNS?

DNS (Domain Name System) ist das „Telefonbuch des Internets": Es übersetzt Hostnamen wie google.com in IP-Adressen. Normalerweise ist ein DNS-Eintrag statisch – DynDNS macht ihn dynamisch: Der Eintrag wird automatisch aktualisiert, wenn sich die IP ändert.

Wie DynDNS funktioniert – Schritt für Schritt

  • Du registrierst einen kostenlosen Hostnamen, z.B. mein-server.ddns.net

  • Auf deinem Router oder Server-PC läuft ein DynDNS-Client (viele Fritz!Boxen haben das eingebaut). Er prüft regelmäßig deine öffentliche IP.

  • Ändert sich die IP, sendet der Client ein Update an den DynDNS-Server: „mein-server.ddns.net zeigt jetzt auf 91.48.77.200"

  • Der DynDNS-Server aktualisiert den DNS-Eintrag (TTL meist sehr kurz: 60–300 Sekunden)

  • Freunde verbinden einfach mit mein-server.ddns.net:25565 – der Hostname bleibt immer gleich

⚠️ TTL – Wie schnell aktualisiert sich DNS?

DNS-Einträge werden zwischengespeichert (TTL = Time To Live, in Sekunden). Bei DynDNS ist die TTL meist 60–300 Sekunden. Nach einer IP-Änderung kann es bis zu ~5 Minuten dauern, bis alle Clients die neue IP sehen. In dieser Zeit schlägt die Verbindung fehl.

Schritt 3 & 4

Firewall & Vollständiges Setup

Selbst mit Port-Weiterleitung und DynDNS kann es noch scheitern – wenn die Firewall auf dem Router oder dem Server-PC den Port blockiert. Es gibt zwei Firewalls, die beide freigegeben sein müssen.

Zwei Firewalls – beide müssen offen sein

🔒 Router-Firewall

Router blockt standardmäßig alle eingehenden Verbindungen. Die Port-Weiterleitung ist gleichzeitig die Freigabe – wer eine Weiterleitungsregel anlegt, öffnet damit automatisch den Port in der Router-Firewall.

🔒 PC-Firewall (Windows/Linux)

Das Betriebssystem hat eine eigene Firewall. Windows fragt beim ersten Start von Minecraft automatisch. Linux muss manuell konfiguriert werden.

Vollständige Setup-Checkliste

  • 1. Minecraft Server installieren – starten mit java -jar server.jar. Server lauscht auf 0.0.0.0:25565
  • 2. Feste private IP – DHCP-Reservierung im Router (per MAC-Adresse) oder statisch am PC: 192.168.1.100
  • 3. Port-Weiterleitung – im Router: TCP 25565 → 192.168.1.100:25565
  • 4. PC-Firewall öffnen – Port 25565 TCP eingehend erlauben (Windows/Linux)
  • 5. DynDNS einrichten – Account anlegen, Client auf Router oder Server-PC konfigurieren
  • 6. Testen – von außen verbinden (z.B. Mobilfunk statt WLAN nutzen!)

Firewall-Befehle

Linux – ufw:
sudo ufw allow 25565/tcp sudo ufw status
Linux – iptables:
iptables -A INPUT -p tcp --dport 25565 -j ACCEPT
Windows – PowerShell (als Admin):
New-NetFirewallRule -DisplayName "Minecraft" ` -Direction Inbound -Protocol TCP ` -LocalPort 25565 -Action Allow

🏠 NAT-Loopback (Hairpin-NAT)

Wenn du dich selbst über den DynDNS-Namen verbindest, schickt dein PC ein Paket an die eigene öffentliche IP. Nicht alle Router können solche Pakete zurück ins LAN leiten (NAT-Loopback). Lösung: Intern immer direkt die private IP 192.168.1.100 oder localhost verwenden.

Paketanalyse

Der vollständige Paketweg

Jetzt schauen wir uns an, wie ein Paket von einem Freund (in seinem eigenen privaten Netz) bis zu deinem Minecraft Server reist – und zurück. Beide Seiten sitzen hinter einem Router.

Unsere Beispielkonfiguration

Freund (Client-Seite)
🖥️ PC: 192.168.2.50
🔌 Quellport: 54321
📡 Router WAN: 85.10.20.5
Server-Seite (du)
⛏️ MC Server: 192.168.1.100:25565
📡 Router WAN: 91.48.77.200
🌐 DynDNS: mein-server.ddns.net

Schritt 1: DNS-Auflösung

Bevor das erste Spielpaket gesendet wird, fragt der MC Client den DynDNS-Server:

Anfrage: mein-server.ddns.net → was ist die IP?
Antwort: 91.48.77.200

Schritt 2: PAT beim Freund-Router (ausgehend)

Der Freund baut die TCP-Verbindung auf. Sein Router macht normales PAT:

AbschnittSourceDestination
Im LAN des Freundes192.168.2.50:5432191.48.77.200:25565
Nach PAT (ins Internet)85.10.20.5:5432191.48.77.200:25565

Der Router des Freundes ersetzt die private Source-IP durch seine öffentliche IP. Da kein Port-Konflikt besteht, bleibt Port 54321 gleich.

Schritt 3: Port-Weiterleitung beim Server-Router

Das Paket kommt an der WAN-Schnittstelle an. Router B prüft: Port 25565 → Weiterleitungsregel!

AbschnittSourceDestination
Ankunft am WAN85.10.20.5:5432191.48.77.200:25565
Nach Port-Weiterleitung85.10.20.5:54321192.168.1.100:25565

Nur die Destination-IP ändert sich – von der öffentlichen Router-IP zur privaten Server-IP. Source und Ports bleiben unangetastet.

Schritt 4: Ankunft am Minecraft Server

Der MC Server empfängt das Paket. Aus Sicht des Servers:

SourceDestination
85.10.20.5:54321192.168.1.100:25565

Der Server sieht die öffentliche IP des Freundes. Er weiß nicht, dass dahinter ein privates Netz mit 192.168.2.50 steckt.

Schritt 5: Der Rückweg (Antwort)

AbschnittSourceDestination
MC Server → Server-Router192.168.1.100:2556585.10.20.5:54321
Server-Router → Internet91.48.77.200:2556585.10.20.5:54321
Freund-Router (PAT rückwärts)91.48.77.200:25565192.168.2.50:54321

Router B ersetzt Source-IP 192.168.1.100 durch seine öffentliche WAN-IP. Router A des Freundes übersetzt Destination zurück zur privaten IP des Freundes.

Der Weg in 5 Sätzen

  1. Der Freund gibt mein-server.ddns.net:25565 ein – DynDNS löst den Namen in die aktuelle öffentliche IP auf.
  2. Sein Router macht PAT: die private IP des Freundes wird durch seine öffentliche IP ersetzt.
  3. Das Paket erreicht Router B auf Port 25565 – die Port-Weiterleitung ändert die Destination auf 192.168.1.100.
  4. Der Minecraft Server empfängt und verarbeitet das Paket und antwortet.
  5. Die Antwort durchläuft beide Router rückwärts und kommt beim Freund an.
